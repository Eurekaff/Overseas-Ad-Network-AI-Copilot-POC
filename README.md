# Overseas Ad Network Embedded AI Assistant POC

中文名称：海外广告联盟业务提效 POC：嵌入式 AI 辅助诊断

## 项目定位

本 POC 选择海外广告联盟中的三个高频业务流程，模拟 AI 如何嵌入广告主后台、开发者后台和内部运营工作台。产品首先呈现业务数据与规则识别出的异常，再由 AI 辅助生成证据绑定的解释、建议动作与沟通内容，并通过人工确认和模拟指标回看验证价值。

它不是独立 AI 平台，也不是聊天机器人产品。AI 是已有业务流程中的辅助能力，业务问题解决优先于模型能力展示。

## 背景与调整原因

原叙事偏向“海外广告联盟 AI 诊断与提效平台”。结合面试反馈，本次调整将重点从搭建单独平台转向真实工作流增强：

- 团队无需为了加入 AI 而建立新的独立产品入口。
- 广告联盟的异常定位、解释和客户沟通已经存在高频且耗时的任务。
- 规则更适合负责可验证的识别和拆解，AI 更适合负责解释、报告和表达。
- 涉及预算、出价、广告源和关键配置的动作仍必须由人工确认。

## 核心业务场景

### 广告主投放看板 + AI 异常解释

广告主或运营人员先查看消耗、展示量、点击量、CTR、CVR、CPA、ROAS、预算利用率、地区和素材表现。页面用规则识别 CPA 上升与 ROAS 下降，再由 AI 解释 CVR 下滑、美国地区和素材影响，并提供客户解释、优化待办、周报和持续观察入口。

### 开发者收益看板 + AI 收益波动分析

开发者或开发者运营人员先查看请求量、填充率、展示率、展示量、eCPM、收入、加载耗时和广告源表现。规则先识别收入与填充率下降，再由 AI 辅助说明巴西激励视频与广告源 B 的影响，提供沟通话术、排查清单、支持工单和 24 小时观察入口。

### 内部运营工作台 + AI 报告辅助

运营、销售和客户成功人员先看到客户列表、重点事项和最近诊断结果，再通过任务模板生成周报、对外邮件、会议纪要和跟进记录。报告输出区分对外话术、内部备注与风险提醒，避免把不确定判断直接对客户输出。

## 设计原则

1. 业务问题优先：先出现看板数据和异常，再出现 AI 能力。
2. 规则先行，AI 解释：规则识别阈值异常并拆解指标，AI 将证据转成自然语言和行动建议。
3. 证据绑定：每项辅助结论都应可追溯到指标、维度或最近诊断结果。
4. 人工确认高风险动作：预算调整、出价修改、广告源切换和关键配置变更只建议，不自动执行。
5. 指标验证价值：使用效率与业务结果指标判断 AI 辅助是否有价值。

## 页面结构

三个业务流程页面均遵循相同结构：

```text
业务场景说明 -> 核心业务指标看板 -> 证据数据
-> 规则识别与指标拆解 -> AI 辅助分析 -> 建议动作
-> 人工确认 / 反馈入口 -> AI 辅助价值衡量
```

首页从“AI 平台首页”调整为“业务流程提效概览”，展示三条嵌入式工作流、处理链路、设计原则和模拟价值指标。

## Mock 数据说明

POC 继续使用本地模拟数据，不接入真实广告平台、SDK 或结算系统。

- 广告主案例：`Galaxy Quest Studio` 的美国 RPG 投放，CPA 上升且 CVR 下滑。
- 开发者案例：`Puzzle Island` 的巴西激励视频广告位，收入与填充率下滑。
- 运营案例：基于上述诊断结果生成客户周报和对外沟通内容。
- 价值指标：人工排查平均 30 分钟，AI 辅助后 8 分钟，建议采纳率 62%，报告人工修改率 35%，客户响应时间节省 45%。以上均明确标注为模拟数据。

## AI 辅助价值衡量

页面展示以下模拟指标，作为产品价值验证框架：

- AI 辅助诊断次数与平均异常排查时间。
- 人工排查耗时和 AI 辅助耗时对比。
- 建议采纳率与报告人工修改率。
- 客户响应时间节省。
- 后续可扩展开发者问题自助解决率。

## POC 边界

- 不接入真实投放、广告 SDK、归因或结算数据。
- 不实现登录、复杂权限、多租户或完整广告投放系统。
- 可使用 mock 输出；本地开发服务保留 Qwen 调用代理作为后续接入预留。
- AI 不直接执行预算、出价、广告源顺序或关键配置修改。

## 本地运行

```bash
npm install
npm run dev
```

如需连接 Qwen 生成真实辅助诊断，可在 `.env.local` 配置本地开发代理：

```bash
QWEN_API_KEY=your_api_key
# 可选：QWEN_MODEL=qwen-plus
# 可选：QWEN_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
```

未配置密钥或生成失败时，界面会明确提示并回退到可演示的模拟辅助结果。

## 代码结构

本轮标准化重构将业务逻辑、模型适配和展示层拆开，便于测试与后续迭代：

- `src/domain/`：模拟数据、诊断规则、报告上下文与纯计算逻辑。
- `src/services/`：浏览器端调用 `/api/qwen-diagnosis` 的请求边界。
- `server/qwen/`：提示词、严格 JSON 解析、Qwen provider 与 Vite 本地中间件。
- `src/app/`、`src/features/`、`src/components/`：应用壳、五个业务页面与共享 UI。
- `src/styles.css`：选定的明亮运营控制台视觉系统与响应式布局。

## 面试展示话术

建议用 5 分钟完成展示：

1. `0:00 - 0:45` 在概览页说明定位变化：不是建立独立 AI 平台，而是在三个既有流程里增加可验证的辅助能力。
2. `0:45 - 2:15` 进入广告主投放页面：先指出 CPA / ROAS 与 CVR 证据，再展示规则、AI 解释、建议动作和人工边界。
3. `2:15 - 3:15` 进入开发者收益页面：用收入下降与填充率拆解证明同一机制可复用于变现场景。
4. `3:15 - 4:15` 进入内部运营工作台：展示诊断结果如何被复用为周报、邮件和跟进动作，体现效率价值。
5. `4:15 - 5:00` 展示闭环追踪和模拟价值指标，总结“规则保证可验证、AI 提高解释效率、人工控制风险、指标验证价值”。
