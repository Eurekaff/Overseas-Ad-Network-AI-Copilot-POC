# Product Requirements

## 1. 产品概述

海外广告联盟业务提效 POC：嵌入式 AI 辅助诊断，是一个面向面试展示的 B2B SaaS 产品原型。它通过模拟广告联盟业务数据，展示 AI 如何嵌入广告主、开发者和内部运营已有流程，辅助异常解释、建议生成、人工确认和后续观察。

第一版不追求完整广告系统能力，重点是让面试官在约 5 分钟主线展示内理解：

- 海外广告联盟的双边平台业务逻辑。
- 广告主投放效果诊断链路。
- 开发者广告变现收益诊断链路。
- 内部运营工作台中的 AI 报告辅助链路。
- 规则判断与 LLM 输出结合的产品设计。
- 产品闭环，而不是单次 AI 回复。

## 2. 目标用户

### 2.1 广告主

广告主是海外手游发行团队，关注买量效果。

核心诉求：

- 解释 CPA 上升原因。
- 解释 ROAS 下降原因。
- 找到影响最大的国家、广告组或素材。
- 获得可执行优化建议。
- 跟踪建议采纳后的指标变化。

### 2.2 开发者

开发者是海外 App / 游戏开发团队，通过激励视频和插屏广告变现。

核心诉求：

- 解释广告收入下降原因。
- 判断是请求量、填充率、展示率还是 eCPM 造成变化。
- 找到影响最大的国家、广告位或广告源。
- 获得配置调整建议。
- 跟踪建议采纳后的收益指标变化。

### 2.3 内部运营 / 销售 / 客户成功

内部团队服务广告主和开发者客户。

核心诉求：

- 快速生成客户周报。
- 快速生成异常复盘。
- 快速生成对外沟通话术。
- 区分对外信息和内部备注。
- 支持中英文沟通。

## 3. 产品原则

### 3.1 不做空聊天框

AI 入口必须绑定具体业务任务。每个核心模块都需要有结构化输入、数据上下文、规则判断和标准输出。

### 3.2 输出必须可验证

AI 报告必须包含指标、变化幅度、影响范围和证据。不能只给出泛泛建议。

### 3.3 规则和 AI 结合

规则负责异常识别和指标拆解，AI 负责解释、建议和沟通表达。

### 3.4 高风险动作保留人工确认

预算调整、广告源配置、投放策略修改等动作在 POC 中只提供建议，不自动执行。

### 3.5 展示产品闭环

平台需要展示从异常发现到后续观察的闭环：

异常发现 -> 诊断报告 -> 建议动作 -> 人工确认 -> 后续观察 -> 效果回看。

## 4. 信息架构

第一版包含以下页面：

- Overview
- Advertiser Diagnosis
- Developer Diagnosis
- Operations Workbench + AI Report Assistance
- Follow-up Tracker

全局能力：

- 角色切换：Advertiser / Developer / Operator
- 语言切换：中文 / English
- 当前模拟数据日期展示

## 5. 功能需求

## 5.1 Overview

### 目标

让用户快速理解当前平台状态和待处理异常。

### 页面内容

- 今日异常数量。
- 已生成 AI 诊断数量。
- 待人工确认建议数量。
- 观察中的指标数量。
- 三类角色入口。
- 闭环流程可视化。

### 交互要求

- 点击角色入口可切换到对应诊断模块。
- 点击待确认建议可进入 Follow-up Tracker。
- 切换语言后，页面内容同步切换。

## 5.2 Advertiser Diagnosis

### 目标

帮助广告主诊断海外手游投放中 CPA 上升和 ROAS 下降的原因。

### 默认业务案例

- 广告主：Galaxy Quest Studio
- 广告计划：US RPG User Acquisition
- 时间范围：近 7 日 vs 前 7 日
- 问题类型：CPA 上升 / ROAS 下降

### 结构化输入

- 广告账户
- 广告计划
- 时间范围
- 问题类型

### 指标展示

核心指标卡：

- Spend
- Impressions
- Clicks
- CTR
- CVR
- CPA
- ROAS
- Budget Utilization

图表：

- CPA 趋势
- ROAS 趋势
- CTR / CVR 趋势

维度拆解：

- 国家表现
- 素材表现

### 规则诊断要求

系统需要判断：

- CPA 是否显著上升。
- ROAS 是否显著下降。
- CTR 是否下降。
- CVR 是否下降。
- 哪个国家贡献最大影响。
- 哪个素材贡献最大影响。

### AI 输出要求

广告主诊断报告必须包含：

- 异常摘要。
- 关键指标变化。
- 主要影响范围。
- 证据数据。
- 可能原因。
- 优化建议。
- 风险提示。
- 后续观察指标。

### 人工确认

每条建议支持状态：

- 已采纳
- 暂不采纳
- 需要人工复核

状态更新后，应能在 Follow-up Tracker 中看到对应记录。

## 5.3 Developer Diagnosis

### 目标

帮助开发者诊断激励视频和插屏广告的收益下降原因。

### 默认业务案例

- 开发者：Puzzle Island Games
- App：Puzzle Island
- 广告位：Rewarded Video - Level Complete
- 国家：Brazil
- 广告源：All Sources
- 时间范围：昨日 vs 前日

### 结构化输入

- App
- 广告位
- 国家 / 地区
- 广告源
- 时间范围

### 指标展示

核心指标卡：

- Requests
- Fill Rate
- Show Rate
- Impressions
- eCPM
- Revenue
- Load Time

图表：

- Revenue 趋势
- Fill Rate 趋势
- eCPM 趋势

维度拆解：

- 国家表现
- 广告位表现
- 广告源表现

### 规则诊断要求

系统需要判断：

- 收入是否显著下降。
- 请求量是否下降。
- 填充率是否下降。
- 展示率是否下降。
- eCPM 是否下降。
- 加载耗时是否异常。
- 哪个广告源影响最大。

### AI 输出要求

开发者诊断报告必须包含：

- 收入下降摘要。
- 指标拆解。
- 影响最大的国家 / 广告位 / 广告源。
- 证据数据。
- 配置建议。
- 风险提示。
- 后续观察指标。

### 人工确认

每条建议支持状态：

- 已采纳
- 暂不采纳
- 需要人工复核

状态更新后，应能在 Follow-up Tracker 中看到对应记录。

## 5.4 Operations Workbench + AI Report Assistance

### 目标

帮助内部运营、销售和客户成功团队快速生成客户复盘、异常解释和沟通内容。

### 入口形式

以任务模板为主，自由输入为辅。

### 任务模板

- 生成客户周报。
- 生成异常复盘。
- 生成对外沟通话术。
- 总结工单并给出处理建议。

### 结构化输入

- 任务模板。
- 客户 / 开发者。
- 关联诊断报告。
- 时间范围。
- 输出语言。
- 可选补充说明。

### 自由输入

自由输入用于补充特殊要求，例如：

- 语气更适合对 CEO 汇报。
- 压缩到 3 个要点。
- 输出英文客户邮件。

自由输入不应替代任务模板。

### 输出要求

内部 Copilot 输出必须包含：

- 客户 / 开发者画像。
- 核心数据变化。
- 异常原因摘要。
- 建议动作。
- 对外沟通话术。
- 内部备注。
- 风险提醒。

### 对外与内部信息隔离

对外沟通话术应避免过度确定和内部判断。

内部备注可以包含：

- 需人工确认的不确定因素。
- 客户风险。
- 跟进优先级。
- 建议内部负责人。

## 5.5 Follow-up Tracker

### 目标

展示建议采纳后的后续观察和效果回看，体现产品闭环。

### 页面内容

建议列表字段：

- 建议标题。
- 来源模块。
- 关联客户 / App。
- 优先级。
- 状态。
- 创建时间。
- 观察窗口。

后续观察字段：

- 指标名称。
- 采纳前数值。
- 当前数值。
- 变化方向。
- 是否改善。

### 状态

建议状态：

- Accepted
- Rejected
- Needs Review

观察状态：

- Monitoring
- Improved
- No Change
- Worsened

### 默认展示案例

广告主侧：

- 建议：检查美国落地页加载速度与转化回传。
- 观察指标：CVR、CPA、ROAS。
- 模拟结果：CVR 回升，CPA 回落，ROAS 小幅改善。

开发者侧：

- 建议：临时提升备用广告源 C 的优先级。
- 观察指标：Fill Rate、Revenue、eCPM。
- 模拟结果：填充率回升，收入部分恢复，eCPM 保持稳定。

## 6. 双语需求

### 6.1 覆盖范围

中英文切换需要覆盖：

- 导航。
- 页面标题。
- 表单字段。
- 指标名称。
- 图表标题。
- 表格字段。
- AI 诊断报告。
- 建议动作。
- 风险提示。
- 后续观察指标。

### 6.2 指标命名

以下指标在中英文中均保留缩写：

- CTR
- CVR
- CPA
- ROAS
- eCPM

中文界面中可以在 tooltip 或说明中展示中文含义。

## 7. LLM 需求

### 7.1 第一版策略

第一版开发可以使用 mock LLM 输出，保证演示稳定。

### 7.2 后续 Qwen 接入

需要预留 Qwen provider：

- Provider 输入为结构化诊断上下文。
- Provider 输出为结构化 JSON。
- 前端不依赖某个具体模型返回的自由文本。
- mock provider 与 Qwen provider 可切换。

### 7.3 LLM 输入上下文

建议包含：

- 当前语言。
- 当前角色。
- 诊断场景。
- 用户选择对象。
- 当前周期指标。
- 基准周期指标。
- 规则诊断结果。
- 证据数据。
- 输出任务类型。

### 7.4 LLM 输出约束

LLM 输出必须符合 `AI_OUTPUT_SCHEMA.md` 中定义的结构。

如果模型输出不完整，系统应能降级使用 mock 或模板化内容。

## 8. 数据需求

### 8.1 数据来源

第一版使用模拟数据。

### 8.2 数据案例

至少需要包含：

1. 广告主 CPA 上升 / ROAS 下降案例。
2. 开发者巴西激励视频收入下降案例。
3. 内部运营周报生成案例。
4. 建议采纳后的后续观察数据。

### 8.3 数据要求

模拟数据应支持：

- 当前周期 vs 基准周期对比。
- 按国家拆解。
- 按素材拆解。
- 按广告位拆解。
- 按广告源拆解。
- 后续观察。

## 9. 非功能需求

### 9.1 演示稳定性

POC 必须优先保证面试演示稳定。

即使 Qwen API 不可用，也应能通过 mock provider 完成完整演示。

### 9.2 信息密度

界面应符合 B2B SaaS 数据平台风格：

- 信息密度适中。
- 图表和表格清晰。
- 不做营销官网式首页。
- 不做纯聊天产品。

### 9.3 可解释性

每个 AI 结论都应能找到对应证据。

### 9.4 安全边界

AI 建议不自动执行任何高风险动作。

## 10. 非目标

第一版不做：

- 真实登录。
- 真实权限。
- 真实广告投放。
- 真实 SDK 接入。
- 真实结算。
- 真实归因。
- 自动预算调整。
- 自动广告源配置。
- 复杂多租户系统。
- 复杂算法训练。
- 完整 CRM。
- 完整工单系统。

## 11. 验收标准

第一版完成后，应能完成以下演示：

1. 在 Overview 中说明平台定位和闭环。
2. 切换到广告主角色，完成 CPA 上升诊断。
3. 展示 AI 诊断报告和证据链。
4. 标记建议状态，并展示后续观察。
5. 切换到开发者角色，完成激励视频收入下降诊断。
6. 切换到内部运营角色，通过任务模板生成客户复盘和对外话术。
7. 切换中英文，页面和报告内容同步切换。
8. 说明当前使用 mock LLM，后续可切换 Qwen provider。
